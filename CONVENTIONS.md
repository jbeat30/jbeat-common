## 1. 개발 철학

- **DX 우선.** 읽기 쉽고, 추적 가능하며, 유지보수하기 쉬운 코드를 우선시한다.
- **기존 공개 동작 보존 우선.** 사용자 경험, 호출 계약, 데이터 흐름을 임의로 변경하지 않는다.
- **실용적 추상화.** 실제 중복과 유지보수 비용이 명확할 때만 로직을 분리한다.
- **라인 수 줄이기보다 추적 가능성과 의도를 우선한다.**
- **막연한 미래 재사용보다 현재 컨텍스트에 맞는 명확한 구조를 선호한다.**
- **영리함이나 과도한 추상화보다 실용적 단순함을 선호한다.**
- **현명해 보이는 코드보다 유지보수 가능한 코드를 목표로 한다.**
- **구현 속도 최적화 전에 회귀 위험을 제어한다.**
- **추측이 아닌 실제 데이터 흐름으로 도달 불가능함이 확인된 경우에만 데드 코드를 제거한다.**

---

## 2. 변경 원칙

- 변경은 요청된 범위 내에서 유지한다.
- 문제를 해결하는 가장 작고 추적 가능한 변경을 만든다.
- 기존 패턴, 네이밍, 흐름, 구조, 에러 처리 스타일을 먼저 존중한다.
- 명확한 정당성 없이 구조 재편, 라이브러리 교체, 파일 이동을 피한다.
- 확실하지 않은 부분이 있을 때는 가정을 명시적으로 밝힌다.

---

## 3. 판단 원칙

- 불확실한 영역에 대해 임의의 결론을 내리지 않는다.
- 안전한 가정이 가능하다면, 가정을 명확히 밝히고 진행한다.
- 검증되지 않은 정보를 사실로 제시하지 않는다.
- 실제로 실행하지 않은 검증을 수행했다고 주장하지 않는다.

---

## 4. 함수와 모듈

- 함수나 모듈당 하나의 핵심 책임을 유지하되, 가독성을 해치는 파편화는 피한다.
- 명확성이 높아질 때는 중첩을 줄이기 위해 예외 케이스를 먼저 처리한다.
- 입력, 출력, 사이드 이펙트 경계를 가시적으로 유지한다.
- nullable 및 optional 입력을 명시적으로 처리한다.
- 같은 레이어 내에서 성공과 실패 표현을 일관되게 유지한다.
- 우회 방식 업데이트보다 공식 업데이트 경로를 선호한다.
- 실제 중복이나 유지보수 비용이 확인된 후에만 추상화한다.
- 나중에 필요할 수 있다는 이유만으로 로직을 분리하지 않는다.
- 공통화가 컨텍스트 이해를 어렵게 만든다면 분리하지 않는다.
- 두 가지 접근법이 동일하게 문제를 해결한다면 더 단순한 방법을 선택한다.
- 단일 분류 개념에 대해 하나의 권위 있는 판별 지점을 유지한다.

---

## 5. 구조와 의존성

- 명확한 유지보수 이점이 있을 때만 구조를 변경한다.
- 코드를 이동하기 전에 추적 가능성과 영향 범위를 평가한다.
- 의존성 방향을 일관되게 유지한다.
- 하위 레벨 컴포넌트가 상위 레벨 정책에 직접 의존하지 않도록 한다.
- 편의를 위해 모듈 경계를 깨지 않는다.
- 순환 의존성과 숨겨진 결합을 피한다.
- 공통 모듈이 단순히 편의를 위해 모든 것이 추가되는 장소가 되지 않도록 한다.

---

## 6. 중복과 공통화

- 같은 규칙이 실제로 반복될 때 공통화를 고려한다.
- 라인 수를 줄이기 위해서만 공통화하지 않는다.
- 컨텍스트나 후속 동작이 다를 경우 유사해 보이는 로직을 분리 상태로 유지한다.
- 추상화가 호출 지점의 이해 비용을 증가시킨다면 추상화를 재고한다.

---

## 7. 안정성과 품질

### 안정성

- 항상 외부 입력과 가능한 실패를 고려한다.
- 에러를 숨기지 않는다. 복구 전략이나 실패 영향을 가시적으로 만든다.
- 정상 경로와 실패 경로를 분리하여 작성한다.
- 변경을 통해 기존 공개 동작을 깨지 않도록 주의한다.

### 보안과 민감 정보

- 코드나 로그에 민감한 정보, 시크릿, 자격증명을 노출하지 않는다.
- 외부 데이터를 신뢰하지 않는다. 검증 또는 방어적 로직을 고려한다.
- 임시 해결책이라도 위험한 우회 구현을 쉽게 허용하지 않는다.

### 검증

- 가능한 경우 정적 분석, 테스트, 빌드, 런타임 확인을 수행한다.
- 검증하지 못한 부분을 명확히 밝힌다.
- 검증 결과를 과장하지 않는다.
- 회귀 가능성이 높은 경로를 우선적으로 검토한다.

## 8. 에러 처리

### 사용자 대면 메시지

- 에러 상수는 하나의 전용 파일(예: `errorMessages.ts`)에 유지한다. 에러 문자열을 인라인으로 하드코딩하지 않는다.
- 메시지는 실제로 발생한 상황을 정확하게 반영해야 한다. 실패 원인이 네트워크 업로드 에러인데 "녹음 실패" 메시지가 나타나면 안 된다.
- 에러 상수는 UI 위치가 아닌 의미적 의미에 따라 분리한다 (예: `VOICE_RECORDING_FAILED` vs `VOICE_SEND_FAILED`).
- 사용자 대면 메시지는 명확하고 비기술적이며 실행 가능해야 한다: 무슨 일이 일어났는지, 사용자가 다음에 무엇을 할 수 있는지.
- 렌더링 환경이 HTML을 지원하는 경우(예: `innerHTML`), 에러 메시지 상수에 `<br>` 줄바꿈을 사용할 수 있다. 상수 정의에 이 가정을 문서화한다.

### 개발자 대면 메시지

- 콘솔 및 로그 메시지에는 에러가 발생한 모듈 또는 함수, 기술적 원인, 관련 컨텍스트를 명시한다.
- 예시 패턴: `[Footer.sendVoiceRecording] S3 업로드 실패 — ${error.message}`
- 이 메시지를 사용자 UI에 노출하지 않는다. 사용자 메시지와 개발자 진단을 철저히 분리한다.
- 서비스를 중단시키는 에러에는 `console.error`, 성능이 저하되지만 복구 가능한 상황에는 `console.warn`, 개발 중 흐름 추적에만 `console.debug`를 사용한다.

## 9. 데이터 분류와 책임

### 레이어별 검증 책임 분리

- 두 레이어가 진정으로 다른 검증 기준을 필요로 할 때만 검증 로직을 레이어별로 분리한다.
- 두 레이어 모두 같은 확인만 필요하다면 공유 함수 하나를 사용한다. 중복하지 않는다.
- 진정한 차이의 예: 업로드 레이어는 MIME 타입 + 확장자가 필요하고, 렌더링 레이어는 확장자만 필요한 경우. 이 경우 분리를 유지한다.
- 두 레이어가 동일한 확인으로 충족된다면, 공유 함수 하나가 올바른 방식이다.

### 단일 권위 분류 함수

- 각 분류 개념은 정확히 하나의 함수가 담당해야 한다. 두 함수가 다른 방법으로 같은 질문에 답한다면 하나를 제거한다.
- 중복 분류기를 제거할 때, 제거된 함수의 다른 동작에 의존했던 호출 지점이 없는지 모든 호출 지점을 추적한다.

## 10. 네이밍 컨벤션

### 기본 원칙

- 역할과 의도를 드러내는 이름을 사용한다.
- 이름이 길어지더라도 약어보다 명확함을 선호한다.
- 단위, 범위, 출처가 중요하다면 이름에 반영한다.
- boolean에는 적절한 경우 `is`, `has`, `can`, `should`, `needs`를 사용한다.
- 배열, 맵, 셋에는 복수형 또는 명확히 집합적인 이름을 사용한다.
- 프로젝트가 해당 패턴을 따른다면 이벤트 핸들러에 `handle`을 사용한다.
- 생성(`create*`), 변환, 조회(`get*`) 함수에는 역할을 드러내는 동사를 사용한다.
- 필요한 경우 외부 필드 이름과 계약을 내부 이름과 별도로 관리한다.

### 외부 데이터

- 외부 시스템 명세를 보존해야 하는 데이터는 내부 이름과 별도로 관리한다.
- 내부 표현이 외부 표현과 다르다면 변환 지점을 명확히 정의한다.
- 혼란을 야기하는 방식으로 외부 계약을 임의로 변환하지 않는다.
- 내부 선호도에 맞추기 위해 외부 필드 이름을 함부로 변경하지 않는다.

### 변수와 상수

- 실제로 재할당이 필요한 경우가 아니라면 값을 불변으로 취급한다.
- 의미가 약한 임시 이름을 피한다.
- 가독성이 향상될 때는 관련 선언을 스코프 상단에 그룹화한다.
- 이름만으로는 출처, 단위, 정책을 설명하지 못할 때만 짧은 후행 `//` 주석을 추가한다.
- 추적 가능성을 높이는 경우 불명확한 리터럴, 임계값, 키, 정책 값을 명명된 상수로 승격한다.
- 정의가 사용 지점에서 너무 멀어져 추적이 어려워진다면 과도한 분리를 피한다.

## 11. 주석 컨벤션

기본적으로 한국어로 작성한다. 전체 언어 정책은 [언어 정책](#12-언어-정책)을 참고한다.

간결하고 일관되며 구현 세부사항이 아닌 의도와 제약에 집중한다.

### 기본 스타일

- 명사형 표현으로 끝낸다: `담당`, `반환`, `수행`, `정의`, `검증`.
- `~을 담당하고 있습니다`, `~하는 로직입니다` 같은 장황한 문장을 피한다.
- **Why** (비즈니스 규칙)와 **What** (핵심 기능)에 집중하고, How는 다루지 않는다.
- 주석은 마침표로 끝내지 않는다.

### 주석 위치

| 위치 | 스타일 | 사용 시점 |
|------|--------|-----------|
| 외부 공개 함수 또는 클래스 | `/** @description ...\n * @param ...\n * @returns ... */` | 공개 함수와 클래스 |
| 인터페이스 또는 타입 필드 | 후행 `//` | 이름만으로 필드 목적이 명확하지 않을 때 |
| 내부 브랜치 또는 블록 | `// WHY 한 줄` | 정책 근거, 비직관적인 조건 |
| 변수 또는 상수 | `const x = ...; // 보충 설명` | 이름만으로 불충분할 때 |
| 내부 private 헬퍼 | `//` 또는 생략 | 이름만으로 충분하지 않을 때만 |

**JSDoc (`/** ... */`)** 은 외부 공개 함수와 클래스에만 사용한다. 파일 헤더나 내부 세부사항에는 사용하지 않는다.
**인라인 `//`** 은 본문 내 모든 주석에 사용한다: 브랜치 근거, 변수 설명, 블록 레이블.
**스타일을 혼용하지 않는다**: 변수에 `/** */`는 잘못된 것. 외부 공개 함수 헤더에 `//`는 잘못된 것.

### 주석이 필요한 경우

- 정책 기반 브랜치 또는 예외 처리
- 외부 시스템 제약 또는 데이터 계약
- 성능 최적화, 우회 처리, 캐싱의 이유
- 이름만으로는 의도를 파악하기 어려운 경우
- 복잡한 상태 변경 또는 사이드 이펙트
- 레거시 데이터 형식을 위한 폴백 로직 (예: `// 레거시 data.url 폴백`)

### 피해야 할 주석

- 코드를 그대로 재서술하는 주석
- 쉽게 구식이 될 수 있는 상태 설명
- 개인적인 인상이나 추측성 메모
- 구현에서 벗어날 긴 절차적 설명
- 구현이 변경될 때 업데이트될 가능성이 낮은 설명

## 12. 언어 정책

### 기본 언어: 한국어

- 프로젝트에서 다른 언어를 요구하지 않는 한 모든 코드 주석을 한국어로 작성한다.
- 모든 작업 결과 보고서와 검증 요약을 기본적으로 한국어로 작성한다.
- 사용자가 명시적으로 요청할 때만 다른 언어를 사용한다.

### 다른 언어를 사용하는 경우

- 사용자가 이 세션 또는 작업에 다른 언어를 명시적으로 요청한 경우
- 프로젝트 로컬 규칙이 다른 언어를 지정한 경우
- 보존해야 하는 특정 언어를 사용하는 외부 시스템 문서 또는 계약

### 적용 범위

이 언어 정책이 적용되는 대상:
- 인라인 주석 (`//`)
- JSDoc 주석 (`/** ... */`)
- 최종 결과 보고서
- 검증 요약
- 사용자에게 보내는 상태 및 진행 메시지

적용되지 않는 대상:
- 코드 식별자 (변수명, 함수명, 클래스명) — 네이밍 컨벤션 따름
- 외부 API 계약, 필드명, 명세
- 파일명 및 디렉토리명

## 13. 품질 체크리스트

### 작업 시작 전

- 요청된 범위를 파악한다.
- 보존해야 할 기존 동작을 파악한다.
- 최소한의 필요한 변경을 정의한다.
- 기존 코드 패턴 및 로컬 저장소 규칙과의 충돌을 확인한다.
- 실행 순서, 초기화 순서, 의존성 순서, 렌더링 순서, 이벤트 흐름이 보존되어야 하는지 검토한다.
- 타입, 린트, 빌드, 런타임, 사이드 이펙트 영향 가능성을 검토한다.
- 지금 검증할 수 있는 것과 없는 것을 구분한다.

### 작업 중

- 변경을 작고, 추적 가능하며, 범위 내로 유지한다.
- 변경할 명확한 이유가 없다면 파일 구조를 보존한다.
- 요청된 변경과 직접 관련이 없다면 기존 네이밍 및 유틸리티 사용을 보존한다.
- 확인된 필요성 없이 추상화를 도입하지 않는다.
- 명확한 정당성 없이 기존 가드, 검증, 폴백 로직, 에러 처리를 제거하지 않는다.
- 사이드 이펙트를 가시적이고 의도적으로 유지한다.
- `any`, 약한 단언, optional-chaining 남용으로 문제를 숨기지 않고 타입 안전성을 유지한다.
- 원래 구조를 약화시키지 않고 린트와 빌드 안정성을 유지한다.
- 명시적으로 변경이 요구되지 않는 한 의미 있는 실행 순서를 보존한다.

### 품질 및 안정성 기준

- 타입 에러를 도입하지 않는다.
- 린트 에러를 도입하지 않는다.
- 빌드 에러를 도입하지 않는다.
- 런타임 에러를 도입하지 않는다.
- 의도하지 않은 사이드 이펙트를 도입하지 않는다.
- 요청이 명시적으로 변경하지 않는 한 기존 공개 동작을 보존한다.
- 요청이 명시적으로 변경하지 않는 한 기존 호출 계약을 보존한다.
- 요청이 명시적으로 변경하지 않는 한 기존 데이터 흐름을 보존한다.
- 요청이 명시적으로 변경하지 않는 한 기존 사용자 경험을 보존한다.
- 정상 경로와 실패 경로가 의도치 않게 변경되지 않도록 한다.

### 검증 규칙

- 가능한 경우 정적 검사, 테스트, 빌드, 런타임 검증을 실행한다.
- 실제로 실행된 검증만 보고한다.
- 실행된 검증과 실행되지 않은 검증을 분리한다.
- 검증하지 못한 항목은 명확히 표시하고 이유를 설명한다.
- 가정과 검증된 사실을 구분한다.
- 검증 범위가 제한될 때는 회귀 위험이 높은 경로를 우선시한다.

### 금지 행동

- 요청된 범위 밖의 대규모 리팩토링을 수행하지 않는다.
- 검증되지 않은 가정을 사실로 제시하고 구현하지 않는다.
- 임시 우회책으로 에러를 숨기지 않는다.
- 저장소 규칙보다 개인 선호를 강요하지 않는다.
- 명시적 요청 없이 공개 동작, 계약, 데이터 흐름, 사용자 경험을 변경하지 않는다.
- 명확한 정당성 없이 파일 이동, 모듈 재편, 라이브러리 교체를 하지 않는다.
- 변경을 성공적으로 보이게 하기 위해 타입 검사, 린트 규칙, 테스트, 빌드 검사를 비활성화하지 않는다.
- 이유와 예상 영향이 명확하지 않으면 순서에 민감한 코드를 변경하지 않는다.
- 데이터 흐름 분석으로 불가능함이 확인된 시나리오에 대한 데드 코드 가드를 추가하지 않는다.

### 최종 자가 점검

- 기존 의도가 보존되었는지 확인한다.
- 변경 범위가 최소한으로 유지되었는지 확인한다.
- 사이드 이펙트가 검토되었는지 확인한다.
- 타입, 린트, 빌드, 런타임 위험이 검토되었는지 확인한다.
- 실행 순서, 초기화 순서, 의존성 순서, 렌더링 순서, 이벤트 흐름이 의도적으로 변경된 경우가 아닌 한 보존되었는지 확인한다.
- 의도적으로 변경된 경우가 아닌 한 공개 동작, 호출 계약, 데이터 흐름, 사용자 경험이 보존되었는지 확인한다.
- 실행된 검증과 실행되지 않은 검증이 별도로 보고되었는지 확인한다.
- 추적 가능성과 유지보수성이 향상되었거나 최소한 저하되지 않았는지 확인한다.
- 이 변경으로 도입되거나 노출된 데드 코드가 제거되었는지 확인한다.
- 에러 메시지가 실제 실패 시나리오를 정확하게 반영하는지 확인한다 (사용자 대면 vs 개발자 대면 분리).
- 분류 로직이 개념당 하나의 권위 있는 함수로 통합되었는지 확인한다.
